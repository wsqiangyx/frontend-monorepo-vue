# Phase 3 产品化发行详细设计

> 制定日期：2026-05-15
> 适用阶段：Phase 3
> 文档性质：模板产品化与发行详细设计
> 上游设计：`docs/总体设计/项目总体设计方案.md`

## 1. 设计定位

Phase 3 只在前两阶段稳定后展开，目标是让仓库具备可交付、可清理、可发布的模板产品化能力。

## 2. 主要范围

- 模板初始化增强
- 模板残留检查增强
- CI 与质量门禁
- 版本与发布治理
- 后端适配层

### 2.1 范围优先级

- P0：初始化、残留检查、CI
- P1：版本与发布
- P1：后端适配
- P2：代码生成、插件与生态扩展

## 3. 不做项

- 不假装已有 `create-*` CLI
- 不绑定固定后端风格
- 不把产品化能力反推回平台基座

## 4. 前置依赖与稳定性假设

本设计假定前序阶段已提供以下稳定基础：

- Phase 0 已提供模板主文档、环境变量样例、初始化脚本入口与统一验证链路
- Phase 1 已提供稳定的平台契约、共享运行时、Mock 替身后端与 Vue3 应用壳
- Phase 2 已提供至少一批可演练的业务样例页面，用于验证模板裁剪后的可运行性

若前序阶段的脚本契约、平台契约或模板交付边界发生破坏性变更，必须同步评估本设计与实施计划。

## 5. 设计原则

- Git 模板优先
- 清理优先
- 发布闭环优先于扩张

### 5.1 初始化边界

Phase 3 可以增强初始化脚本，但不应在文档中把它表述成已经成熟的 `create-*` CLI。

### 5.2 适配边界

真实后端接入必须通过适配层映射到现有平台契约，而不是反向推翻 `shared-service` 与 Mock 的正式模型。

例如：

- 后端返回 `{ userId: 1, user_name: 'admin' }`
- 适配层应先转换为平台内部约定的数据结构（如 `{ id: 1, username: 'admin' }`）
- 再交给 `shared-service`、页面服务层或共享类型消费

## 6. 验收标准

- 模板可初始化
- 残留可检查
- CI 可验证
- 发布链路可扩展

进一步验收建议：

- 初始化后仍能通过 `pnpm verify`
- 模板残留检查可发现 scope、持久化 key 与仓库名残留
- 多平台 CI 至少覆盖 `ubuntu-latest` 与 `windows-latest`

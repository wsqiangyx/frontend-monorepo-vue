# 模板初始化与裁剪指南

本文档面向“第一次把这个仓库改造成自己项目”的使用者。

当前模板定位是：

- 以 Git 模板仓库为主线
- 正式技术栈为 React-only
- 尚未提供 `create-*` CLI
- 当前已提供最小初始化脚本与模板残留检查脚本

所以本文档回答的是：拿到仓库后，第一天怎么安全接手。

## 1. 先做三个决定

### 决定 A：是否立即清理遗留项

当前仓库快照里可能仍包含遗留目录、旧脚本和旧文档入口。

当前已知需要单独判断的目录包括：

- `apps/react-screen-designer`：当前工作区存在的可视化专题子应用预研 / 占位目录，尚未进入根级默认验收链路

你有两个选择：

1. 先保留这些残留，但明确不再继续维护。
2. 立刻做一次专门的清理任务统一删除。

推荐默认值：`先保留并标记为待清理项`。

原因：

- 当前正式契约已经是 React-only，不需要再额外考虑旧技术栈才能启动项目
- 第一日同时做命名替换、CI 接管、残留清理，回归成本更高
- 先跑通 React 基线，再做专门清理，更容易验证影响面

### 决定 B：技术标识现在改不改

这里的“技术标识”主要指：

- 包 scope：`@repo/*`
- 主题持久化 key：`repo-theme-preference`
- 主题样式 key：`repo-theme-style`
- locale 持久化 key：`repo-locale`

推荐默认值：`第一天先不改`。

原因：

- 当前没有完整自动初始化 CLI
- 这些名字分散在共享运行时、别名契约、文档和测试里
- 如果你同时还要接管 CI、改品牌文案、清理历史残留，重命名技术标识会显著抬高风险

### 决定 C：CI 分支模型沿不沿用现状

推荐默认值：`复制仓库后立刻人工复核 CI 配置`。

原因：

- 当前仓库带了工作流文件，但不应假设它天然匹配你的默认分支
- 在你没有核对触发分支、仓库变量和 required checks 之前，不要把它当成“已经完成接管”

## 2. Day 0 推荐顺序

### Step 1：复制模板并建立自己的仓库

任选一种：

- 通过 GitHub Template 创建新仓库
- clone 现有仓库后改 remote

### Step 2：保留一份原始基线快照

建议在第一次大改前打一份基线 tag 或建一个只读分支，用于后续回看：

- 你改了哪些模板残留
- 你清理了哪些遗留目录
- 你有没有误伤共享层契约

### Step 3：决定是否先执行最小初始化脚本

如果你已经明确了项目名、展示名、app 标题和持久化 key 前缀，可以先运行：

```bash
pnpm template:init -- --root . --project-name acme-starter --display-name "Acme Starter" --storage-key-prefix acme-console --app-title "Acme Control Center"
```

这一步的边界很明确：

- 它只做最小范围的文本替换
- 它不会帮你重命名 `@repo/*`
- 它不会自动清理遗留项
- 运行后仍然要人工复核 diff

### Step 4：先改业务可见层

先处理这些低风险内容：

- 仓库显示名
- README 标题与项目介绍
- React 应用名称、页面标题、品牌文案
- 图标、演示图片、默认首页内容
- 团队治理说明

### Step 5：拷贝环境变量样例

根目录有一份变量清单：

- `/.env.example`

正式 app 的运行时样例：

- `apps/react-app/.env.example`

按需复制到 app 目录下，例如：

```bash
Copy-Item apps/react-app/.env.example apps/react-app/.env.local
```

当前示例变量：

```env
VITE_ENABLE_MSW=false
```

如果要启用开发态 mock，再改成 `true`。

### Step 6：安装依赖并先跑一次基线

```bash
pnpm install
pnpm template:check
pnpm verify
```

先确认模板在你的仓库副本里能跑通，再开始清理历史残留或重命名。

## 3. 模板残留检查清单

第一次接手时，建议至少先运行：

```bash
pnpm template:check
```

然后再人工搜索以下关键词：

- `frontend-monorepo`
- `Workspace Shell`
- `@repo/`
- `repo-theme-preference`
- `repo-theme-style`
- `repo-locale`

这些关键词并不都必须当天清零。

推荐处理方式：

- 品牌文案、仓库标题、首页展示类内容：尽快替换
- `@repo/*` 和 `repo-*` 运行时标识：延后到专门任务
- `react-screen-designer` 相关目录与文档：先确认你是否真的要把它纳入正式应用规划，再决定保留、补齐或清理
- 遗留引用：要么统一清理，要么明确标记为待清理
- CI 触发分支和 required checks：在正式依赖 CI 前完成复核

## 4. React-only 收口策略

### 4.1 正式主线

当前正式主线只有：

- `apps/react-app`
- `packages/shared`
- `packages/platform-core`
- `packages/ui-tokens`
- `packages/resources`
- `packages/mock`
- `packages/ui-react`

当前工作区另有：

- `apps/react-screen-designer`

但它现在仍属于专题子应用目录，而不是模板默认正式出口。除非你同步补齐根脚本、根测试矩阵与主文档契约，否则不要把它当成“已经接管完成的第二个正式应用”。

当前治理状态请同步查看根 `STATUS.yaml`。如果某个目录处于 `experimental`，不要把它直接写进你的默认 CI、默认构建和默认交付范围。

后续任何新增方案、组件、规范或页面，都默认围绕这条主线展开。

### 4.2 遗留项的最小检查面

如果你决定清理遗留项，至少同步检查：

- 根 `package.json`
- 根 `vitest.config.ts`
- `tsconfig.packages.json`
- `eslint.config.js`
- 根 `README.md`
- 根 `AGENTS.md`
- 根 `TEMPLATE.md`
- `docs/总体设计/项目总体设计方案.md`
- `docs/专题文档/设计/*`
- `docs/专题文档/计划/*`
- `docs/规范/测试规范.md`

清理后必须执行：

```bash
pnpm verify
```

### 4.3 什么时候不建议立刻清理

出现以下情况时，建议先保留残留、不要第一天立刻删：

- 你还打算同时重命名 `@repo/*`
- 你还没核对 CI 和工作流
- 你还没跑通第一次 `pnpm verify`
- 你还没确认主题、i18n 和资源层的正式边界

## 5. 技术标识重命名建议

如果你确实需要把 `@repo/*` 或 `repo-*` 标识改成自己的命名，请按一个独立任务处理，不要和 Day 0 混在一起。

建议顺序：

1. 先全文检索出现位置。
2. 先改文档与命名策略，再改代码与配置。
3. 一次性改完包名、alias、持久化 key、引用路径。
4. 最后统一回归 `pnpm check:alias`、`pnpm test`、`pnpm build`、`pnpm verify`。

尤其要注意：

- `@repo/shared/routes`
- `@repo/shared/routes/react`
- `@repo/ui-tokens/theme`
- `repo-theme-preference`
- `repo-theme-style`
- `repo-locale`

## 6. CI 接管建议

因为当前模板产品化还在持续推进，复制仓库后不要把 CI 当黑盒。

至少核对：

- 工作流触发分支是否匹配你的默认分支
- 保护分支和 required checks 是否已设置
- 是否需要补 `CODEOWNERS`、PR 模板、发布清单
- 是否还有仍面向旧结构的提示文案需要一并清理

## 7. 首次接手的完成标准

以下条件满足后，才算真正完成了第一次模板接管：

1. 你已经确认仓库正式技术栈为 React-only。
2. 根 README、TEMPLATE、初始化指南已按你的项目事实更新。
3. React app 的 `.env.local` 或等价文件已建立。
4. 你已经跑通 `pnpm install` 与 `pnpm verify`。
5. 你知道哪些标识被保留，哪些被重命名。
6. 你已经人工复核 CI 分支模型与自动触发条件。
7. 你已经决定遗留项是立即清理，还是明确挂入后续任务。

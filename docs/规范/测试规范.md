# 前端 Monorepo 测试规范

## 1. 文档定位

本文档描述当前仓库已经落地的测试基线，以及新增 workspace 时应遵守的测试约束。

- 它回答"测试体系现在应该如何统一执行"
- 它不重复维护总体设计主文档中的架构边界
- 若测试架构正式变更，需要同步更新根 `README.md`、根 `AGENTS.md` 与总体设计主文档
- 本文档只描述已经落地的测试契约与明确识别的治理缺口，不假装未实现能力已经存在

## 2. 当前测试架构

### 2.1 根配置职责

根 `vitest.config.ts` 只负责聚合 workspace 项目，使用 `defineConfig`。

正式测试矩阵覆盖：

- `packages/shared-utils/vitest.config.ts`
- `packages/shared-i18n/vitest.config.ts`
- `packages/shared-service/vitest.config.ts`
- `packages/design-tokens/vitest.config.ts`
- `packages/shared-ui/vitest.config.ts`
- `packages/shared-workflow/vitest.config.ts`
- `apps/vue3-app/vitest.config.ts`

### 2.2 项目级配置职责

每个 workspace 自己的 `vitest.config.ts` 负责项目级配置，统一使用 `defineProject`。

当前基线约束：

- 统一启用 `globals: true`
- 统一使用 `pool: 'threads'`
- alias、插件链与测试环境由各 workspace 自己声明
- app 级 alias 统一从 `paths.config.ts` 派生

## 3. 当前环境矩阵

| Workspace               | 环境    | 说明                              |
| ----------------------- | ------- | --------------------------------- |
| `@repo/shared-utils`    | `node`  | 通用工具函数测试                  |
| `@repo/shared-i18n`     | `node`  | 国际化运行时与语言包测试          |
| `@repo/shared-service`  | `node`  | 服务层、权限、Mock handlers 测试  |
| `@repo/design-tokens`   | `node`  | 设计令牌、主题映射与 CSS 变量测试 |
| `@repo/shared-ui`       | `jsdom` | Vue3 UI 组件测试                  |
| `@repo/shared-workflow` | `node`  | 工作流引擎类型与契约测试          |
| `@repo/vue3-app`        | `jsdom` | Vue3 app 集成测试                 |

根级脚本测试补充：

- `scripts/__tests__/*.test.mjs` 通过 `node:test` 执行
- 它们不属于根 `vitest.config.ts` 的 workspace projects
- 正式入口是根 `pnpm test:scripts`

## 4. 测试分层策略

当前正式测试策略分为四层：

- 单元测试：`shared-utils`、`shared-i18n`、`shared-service`、`design-tokens`
- 组件测试：`shared-ui`
- 集成测试：`vue3-app` 与 `shared-service/mock` 的交互链
- E2E 测试：当前尚未纳入正式默认基线

当前正式要求：

- 单元测试优先验证纯规则、纯函数、共享契约与平台模型
- 组件测试优先验证可见行为、交互结果与可访问性语义
- 集成测试必须通过 `@repo/shared-service/mock` 的正式入口接入替身数据
- 不得在宿主测试中绕过 `@repo/shared-service/mock` 自建另一套 API 替身
- 推荐通过全局 setup 统一接入 `@repo/shared-service/mock/server`，减少各测试文件自行绕过的机会

## 5. 配置约定

### 5.1 Vue3 app

Vue3 app 测试配置应满足：

- 使用 `@vitejs/plugin-vue`
- 使用 `jsdom`
- 复用 app 自己的 alias 配置
- 通过 `src/test/setup.ts` 接入测试初始化

当前 Vue3 app `setup.ts` 的职责包括：

- 启停 `@repo/shared-service/mock/server`
- 每个用例后 `server.resetHandlers()`

### 5.2 共享包

共享包按类型分为两类：

- 运行时包：`shared-utils`、`shared-i18n`、`shared-service`、`design-tokens`
- 框架型共享包：`shared-ui`

对应约束：

- 运行时包优先使用 `node`
- 框架型共享包允许使用 `jsdom`
- 无论哪类 package，都不应把 app 级页面、路由入口或业务状态拉进测试基线

### 5.3 覆盖率

当前仓库保留 `test:coverage` 脚本契约，但未统一设置覆盖率阈值，也未把 `test:coverage` 纳入根 `verify` 的默认门禁。

当前稳定约束只有三条：

- 根与各 workspace 都保留 `test:coverage`
- 覆盖率提供者统一使用 `@vitest/coverage-v8`
- 若某个 workspace 需要覆盖率阈值，必须在自己的 `vitest.config.ts` 中显式声明

## 6. 测试文件规则

### 6.1 命名与位置

- 扩展名使用 `.test.ts` 或 `.test.tsx`
- Vue 组件测试使用 `.test.ts` 配合 `@vue/test-utils`
- 默认放在 `src/__tests__/`
- 允许与组件或模块就近放置，只要与当前 workspace 既有风格一致

### 6.2 构建排除

所有 build 用 `tsconfig.build.json` 都应排除测试文件。

至少应覆盖：

- `src/**/*.test.ts`
- `src/**/*.test.tsx`
- `src/test/**`

## 7. 编写规则

### 7.1 命名风格

- `describe` 描述被测单元
- `it` 描述预期行为
- 测试名优先清晰、可搜索、一致

### 7.2 断言风格

- 统一使用 `expect(...)`
- DOM 场景优先断言用户可见结果或可访问性语义
- 能用语义查询时，优先 `getByRole`、`getByText`

### 7.3 `vitest` API 导入

当前仓库已启用 `globals: true`，但不强制删掉所有 `describe` / `it` / `expect` 显式导入。

实际约束是：

- 可以使用全局注入
- 也允许显式导入
- 新增测试时优先跟随所在 workspace 的既有风格，不要在同一文件混用两套写法
- `vi` 仍显式导入

## 8. Mock 与 MSW 规范

### 8.1 统一入口

MSW 统一通过 `@repo/shared-service/mock/server` 与 `@repo/shared-service/mock/browser` 暴露。

禁止：

- 在 app 测试中直接自建 `setupServer`
- 在业务代码中绕过 `@repo/shared-service/mock/*` 直接引用内部实现路径

### 8.2 生命周期管理

app 层测试初始化统一在各自 `src/test/setup.ts` 中维护。

最低要求：

- `beforeAll -> server.listen()`
- `afterEach -> server.resetHandlers()`
- `afterAll -> server.close()`

### 8.3 测试替身原则

- 平台主链路的测试替身统一收敛到 `@repo/shared-service/mock`
- Vue3 app 集成测试不得直接改写全局 `fetch` 以跳过 `@repo/shared-service/mock/server`
- 若新增数据域，优先先补 `handlers`，再补页面测试

## 9. 当前治理缺口

以下能力已被识别，但当前尚未纳入正式默认基线：

- Playwright / Cypress 等 E2E 测试基线
- 架构规则脚本，例如 `check:arch`
- 文档一致性脚本，例如 `docs:check`
- 防止宿主测试绕过 `@repo/shared-service/mock` 的自动化规则

这些能力落地后，必须同步总体设计主文档与本规范。

## 10. 运行方式

### 10.1 根命令

- `pnpm test`
- `pnpm test:scripts`
- `pnpm test:watch`
- `pnpm test:coverage`

### 10.2 单 workspace

```bash
pnpm -F @repo/vue3-app test
pnpm -F @repo/vue3-app test:coverage
pnpm -F @repo/shared-utils test:watch
```

### 10.3 脚本契约

所有 workspace 测试脚本继续走包装脚本：

```json
{
  "scripts": {
    "test": "node ../../scripts/vitest.cjs run",
    "test:watch": "node ../../scripts/vitest.cjs",
    "test:coverage": "node ../../scripts/vitest.cjs run --coverage"
  }
}
```

不要回退到裸 `vitest`。
